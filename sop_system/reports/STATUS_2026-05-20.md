## 三、当前问题 — Phase 3 回放验证卡住 ⚠️

### 3.1 问题现象
**文件**: [tools/replay_realtime_pipeline_v2.py](tools/replay_realtime_pipeline_v2.py)

用原始 .avi 视频逐帧跑完整管线（YOLO→MediaPipe→90维特征→TemporalPredictorV2→FSM），20个OK视频只通过 **15/20 (75%)**，离线评估是 **98.9%**。

失败视频的FSM路径:
```
ok_8:  [1,7,3,4,5,6]   ← 漏了S2 (放耳机)
ok_9:  [1,2,3,7,5,6]   ← S3→ERROR→S5  (漏S4)
ok_10: [1,2,7,4,5,6]   ← 漏了S3 (放插头)
ok_11: [1,7,3,4,5,6]   ← 漏了S2 (放耳机)
ok_14: [1,7,2,3,4,5,6] ← 先报ERROR才恢复S2
```

### 3.2 根因分析（确定，不是猜测）

**问题不在FSM，不在EventDetector，而在 `TemporalPredictorV2` 的增量预测与离线 `sliding_window_inference()` 行为不一致。**

具体差异:

| 环节 | 离线 eval (98.9%) | 实时 replay (75%) |
|------|-------------------|-------------------|
| 推理方式 | 一次性拿到全部特征，批量滑动窗口 | 逐帧到来，增量滑动窗口 |
| 未覆盖帧处理 | 所有帧都有窗口覆盖后统一fill | 早期帧(0-11)没窗口覆盖→fallback到远处帧 |
| EMA平滑 | 在整个序列上逐类 EMA | 增量EMA，初始状态不同 |
| 预测结果 | 连续稳定的序列 1→2→3→4→5 | 边界处跳变，偶尔跳过步骤 |

**关键代码路径** ([temporal_predictor_v2.py:135-168](engine/temporal_predictor_v2.py#L135-L168)):

`get_frame_prediction()` 有一个 nearest-neighbor fallback:
```python
if frame_idx not in self._frame_probs:
    for d in range(1, 20):  # 向前后搜索最多20帧
        if frame_idx + d in self._frame_probs:  # 优先向前
            probs = self._frame_probs[frame_idx + d] / ...
            break
```

在实时模式下，buffer的前11帧（half_w=12）还没有任何窗口覆盖，`_frame_probs` 为空 → fallback 把后面帧的概率赋值给前面帧 → **时序错位** → 模型瞬间预测S3而不是S2 → FSM看到S1→S3跳步 → MISSING_STEP error (step=7)。

### 3.3 之前走过的弯路

1. **第1版**: 加了 `MinimalEventDetector` 做 confirm_frames=4 + phys_ok 检查 → phys_ok 阻止了S1（因为盒子打开检测延迟）→ FSM路径全是7
2. **第2版**: 去掉 phys_ok (always True)，加上 idle 处理 → 好了些但还有30%失败
3. **第3版**: 删掉 `MinimalEventDetector` 整个类，直接用模型预测 → 还是有25%失败

每次修改都在FSM/EventDetector层面打补丁，没有触及真正的根因：**预测器增量模式 ≠ 离线批量模式**。

## 四、解决方案（两个方案，推荐方案A）

### 方案A（推荐）: 回放时先缓冲全部特征，再批量推理

修改 `replay_realtime_pipeline_v2.py` 的 `process_video()`:
1. 第一遍遍历视频：只做 YOLO+MediaPipe+特征提取，把全部90维特征存入数组（就像离线 `extract_features_v2_90.py` 做的）
2. 特征提取完后：用 `sliding_window_inference()` 一次性批量推理（和离线eval完全一样的函数）
3. 拿到 `preds` 和 `confs` 数组后：逐帧喂给FSM

**优点**: 保证和离线评估 100% 一致的预测结果，98.9% OK通过率可以重现  
**缺点**: 不是真正的"实时"推理，但对回放验证来说是正确的做法  
**注**: 真正的实时推理（摄像头模式）需要方案B，但现阶段回放验证应该先用方案A确认管线正确性

### 方案B: 修复增量预测器

修改 `TemporalPredictorV2`:
1. 对未覆盖帧返回 None（不使用 fallback）
2. replay 脚本中，如果 `get_frame_prediction()` 返回 None，FSM 保持当前状态不变
3. 确保增量 EMA 与批量 EMA 在数学上等价

**优点**: 真正实时  
**缺点**: 复杂度高，需要数学验证

### 我的建议

**先用方案A验证管线正确性**（改动小，确定性高），确认回放能达到98.9%后，再考虑方案B做真正的实时推理。

## 六、总结

**一句话**: v2_90 模型本身没问题（离线98.9%），实时回放只有75%是因为预测器的增量推理方式和离线批量推理不一致，导致早期帧预测错位跳步。修复方法是用批量推理替代增量推理来做回放验证。

**不应该做的事**:
- 不要再改 FSM（sop_fsm.py）
- 不要再加 EventDetector
- 不要重新训练模型
- 不要改特征维度
- 不要放宽 FSM 的 S3/S4 顺序

**应该做的事**:
1. 按方案A修改 replay 脚本，批量推理 → 验证能达到98.9%
2. 验证通过后，再按方案B改造预测器做真正的实时推理
3. 然后进入阶段5（真实摄像头测试）
