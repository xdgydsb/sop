### ❌ 已知问题 / 技术债

1. **idle-skip hack** (`camera_smoke_test_v2.py:442-453`)
   ```python
   if pred_step == 0:  # 模型输出S0
       if self.fsm.current_step == SOPStep.S5_CLOSE:
           self.fsm.current_step = SOPStep.COMPLETE  # 硬跳到完成
   ```
   模型是一个**步骤分类器**(S0-S5)，S0代表"没有动作"，S6(COMPLETE)不是模型输出类别。当人在做S5(关盒子)后模型输出的S0(无动作)被硬编码为"流程完成"。这是临时方案，如果人真的在S2之后停顿(模型出S0)，也会被误判为完成。

2. **特征提取是定频的，不是事件驱动的** 
   — 提取器每1/15秒提取一次特征，无论画面有没有变化。这意味着：
   - 中间帧的细微动作可能被合并到同一个特征窗口
   - 快速完成两个步骤时，第二个步骤的特征可能被稀释

3. **TemporalPredictor的延迟** 
   — T=48窗口 @ 15fps = 3.2秒时间窗口
   — 中心帧延迟 = half_w/15 = 12/15 ≈ 0.8秒
   — 加上特征提取间隔和模型推理时间，实际端到端延迟约 1.0-1.5秒

4. **物理状态模块未接入**
   — `engine/physical_state.py` 已实现但 `camera_smoke_test_v2.py` 里 `physical_state_ok=True` 写死了
   — 这意味着"盒子是否打开"这种物理约束没有真正用来过滤错误预测

5. **从未跑过端到端测试**
   — 所有验证都是"代码可以import"级别
   — 实时推理的帧率、延迟、内存使用、模型精度完全未知
   — 8次烟雾测试(3 OK + 2乱序 + 2漏步 + 1重复)尚未执行

## 二、当前架构流程图

```
海康相机 (MV-CS050-10UC, USB3, BayerRG8, ~30fps)
  │
  ▼ convert_frame_to_bgr()
  彩色帧 (BGR, 2448×2048 或配置分辨率)
  │
  ├─→ YOLO model (yolo_final_v1.pt, CPU) ─→ [5类检测 + Box state + ByteTrack]
  ├─→ MediaPipe Hands ─→ [21点关键点 × 双手]
  │
  ▼ (每 1/15 秒抽取一次)
  FeatureExtractorV2Standalone.extract()
  → 90维特征向量
  │
  ▼
  TemporalPredictorV2.predict(feat)
  → 累积窗口 [t-47, t] → TCN+BiGRU → step_logits (6类: S0-S5)
  │
  ▼
  get_latest_mature_prediction()
  → {step, confidence, top3, step_probs}  ← 只有成熟帧才返回
  │
  ├─→ pred_step == 0?
  │    ├─ 当前在S5_CLOSE → 硬编码跳到COMPLETE
  │    └─ 当前不在S5 → 忽略(保持当前步骤)
  │
  └─→ pred_step != 0 → FSM.validate(pred_step, confidence)
       ├─ 合法转移 → 步骤推进
       ├─ 非法转移 → WRONG_ORDER / MISSING_STEP 错误
       └─ 低置信度(<0.4) → 保持当前状态
```

## 三、拟行动指南（按优先级）

### 第一阶段：验证基本链路可用（预计 1-2 小时）

**目标**: 确认代码能跑，发现并修复启动阶段的硬错误。

1. **执行最小化冒烟测试**
   ```
   cd d:\shabi\sop_system
   C:\Users\HP\AppData\Local\Programs\Python\Python310\python.exe tools\camera_smoke_test_v2.py --imgsz 320
   ```
   - 先用 `--imgsz 320` 降低YOLO输入分辨率，确保CPU能跑起来
   - 只做一个动作(S1开盒子)，看FPS多少，预测是否出现
   - 记录首次运行的错误日志

2. **关键指标观察**
   - 实时帧率(FPS): 期望 ≥ 8fps，能跑就行
   - 特征提取帧率: 15fps 是否稳定
   - 首帧成熟延迟: 是否在 2秒内
   - 内存: 是否稳定不增长

3. **可能遇到的问题和应对**
   | 症状 | 可能原因 | 应对 |
   |------|---------|------|
   | FPS < 3 | YOLO在CPU太慢 | 降低 imgsz 到 256 或 224 |
   | 内存持续增长 | ByteTrack tracker未清理 | 加 tracker cleanup |
   | 始终无预测输出 | 模型加载失败或特征维度错误 | 检查 input_dim=90 |
   | YOLO不检测任何物体 | 相机画面太暗/角度问题 | 调整曝光/确认物体可见 |

### 第二阶段：8次烟雾测试（预计 2-3 小时）

**目标**: 获取完整的端到端性能数据，尤其是错误检测能力。

**测试执行** (按 `s` 开始，`r` 重置，`q` 退出):

| 测试 | 动作序列 | 期望FSM path | 通过标准 |
|------|---------|-------------|---------|
| test_001 OK | S1→S2→S3→S4→S5 | [1,2,3,4,5,6] | COMPLETE, 0 errors |
| test_002 OK | S1→S2→S3→S4→S5 | [1,2,3,4,5,6] | COMPLETE, 0 errors |
| test_003 OK | S1→S2→S3→S4→S5 | [1,2,3,4,5,6] | COMPLETE, 0 errors |
| test_004 乱序 | S1→S3(跳充电器) | [1, ERROR] | MISSING_STEP detected |
| test_005 乱序 | S3→S2(先充后耳) | [ERROR] | WRONG_ORDER detected |
| test_006 漏步 | S1→S2→S4(跳S3) | [1,2, ERROR] | MISSING_STEP detected |

**动作要慢、要清晰**: 每个动作 2-3 秒完成，动作之间有 1 秒停顿。快的动作时序模型来不及反应。

### 第三阶段：根据结果决定方向（预计需要分析后才能定）

**场景A**: 3个OK测试全部通过（FSM path = [1,2,3,4,5,6], 0 errors）
- → 时序模型+FSM基本可用
- → 重点优化: 降低延迟、提升错误检测准确率
- → 可以考虑接入物理状态模块(`physical_state.py`)增强鲁棒性

**场景B**: OK测试能检测到步骤但顺序不对或有漏
- → 需要分析 temporal_trace.json 看模型实际输出
- → 可能原因: 特征提取频率不对、窗口T=48太大、模型过拟合训练数据
- → 对策: 调整 T/stride/特征频率，或考虑用更简单的投票逻辑替代FSM

**场景C**: 完全检测不到步骤或随机输出
- → 模型在真实相机画面上的泛化能力不足
- → 需要收集相机数据重新训练/微调
- → 可能原因: 训练数据是手机拍摄，相机画面分布不同

**场景D**: 相机取流不稳定或FPS太低无法使用
- → 降级方案: 用OpenCV VideoCapture + USB相机通用驱动
- → 或降低相机分辨率来提升FPS

### 第四阶段：性能优化（等基本验证通过后）

1. **YOLO 推理加速**
   - 当前 `device="cpu"`，如果机器有NVIDIA GPU，改 `device="cuda"`
   - 降低 imgsz (320→256→224)，在CPU上能显著提速

2. **特征提取频率自适应**
   - 当前固定15fps，可以在相机帧率高时提升到20fps
   - T=48 窗口时间 = 48/fps，提升fps能降低延迟

3. **修复 idle-skip hack**
   - 方案A: 用时间维度判断 — 在S5之后连续N帧无动作→完成
   - 方案B: 加入物理状态 — 盒子关闭 + 所有物体已在盒内 → 完成
   - 方案C: 模型增加S6类重新训练（成本高）

4. **接入物理状态模块**
   - `engine/physical_state.py` 可以判断: 盒子开/关、物体在盒内/外
   - 用来过滤不可能的动作（盒子关着不可能放东西进去）

## 四、不做的/不该做的事

1. **不要重新训练模型** — 在没有相机实测数据之前，不知道现有模型到底差在哪里，盲目重训是浪费
2. **不要优化FSM规则** — 同上，没有实测数据前调FSM阈值是纸上谈兵
3. **不要在服务器上跑** — 目前阶段全部在本地Windows+Python3.10上验证，服务器没有相机
4. **不要增加复杂性** — 不接入物理状态、不加入HMM滤波、不加新模块，先把最简单的链路跑通
5. **不要追求高FPS** — 先追求"能跑"，再追求"跑得快"
