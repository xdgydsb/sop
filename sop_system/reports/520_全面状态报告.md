## 三、当前问题 — Phase 3 回放验证（⚠️ 卡住）

### 3.1 做了什么

创建了 [tools/replay_realtime_pipeline_v2.py](tools/replay_realtime_pipeline_v2.py)，目标是用原始 .avi 视频完整跑一遍管线：

```
.avi帧 → YOLO检测 → MediaPipe手部 → 90维特征提取 → 时序模型预测 → FSM验证 → 输出结果
```

**不允许**读取预提取的 .npz 特征文件，必须从原始视频逐帧跑。

### 3.2 结果

服务器上跑了 20 个 OK 视频：**15/20 通过 = 75%**

而离线评估是 **179/181 = 98.9%**。差距巨大。

5个失败视频的 FSM 路径：
```
ok_8:  [1, 7, 3, 4, 5, 6]   → 模型预测跳过了 S2
ok_9:  [1, 2, 3, 7, 5, 6]   → 模型预测跳过了 S4
ok_10: [1, 2, 7, 4, 5, 6]   → 模型预测跳过了 S3
ok_11: [1, 7, 3, 4, 5, 6]   → 模型预测跳过了 S2
ok_14: [1, 7, 2, 3, 4, 5, 6] → 模型先跳S3(报错)再恢复S2
```

共同特征：**模型预测跳过了某个中间步骤**（S2或S3），FSM 检测为 MISSING_STEP → step=7(ERROR)。

### 3.3 走过哪些弯路

| 版本 | 改动 | 效果 | 教训 |
|------|------|------|------|
| 第1版 | 加 MinimalEventDetector + phys_ok | 全坏，FSM路径全是7 | EventDetector 的 phys_ok 时序不对，S1的盒子打开检测滞后于模型动作识别 |
| 第2版 | 去掉 phys_ok，只保留 idle 处理 | 好了些，30%失败 | 改错方向了，问题不在 FSM |
| 第3版 | 删除 MinimalEventDetector，直接用模型预测 | 还是 25% 失败 | 问题根本不在 EventDetector |

**核心教训：每次都在 FSM/EventDetector 层面打补丁，但没有触及真正的根因。**

### 3.4 真正的根因

**问题出在 `TemporalPredictorV2` 的增量推理和离线 `sliding_window_inference()` 的批量推理不一致。**

这是两个函数的对比：

| 对比维度 | 离线 eval（98.9%） | 实时 replay（75%） |
|----------|-------------------|---------------------|
| 函数 | `sliding_window_inference()` | `TemporalPredictorV2.predict()` + `get_frame_prediction()` |
| 特征输入 | 一次性拿到全部 N 帧 | 逐帧到来，buffer 逐步增长 |
| 窗口调度 | 从 start=0 到 N-T，stride=4，一次跑完 | 每次只跑新完成的窗口 |
| 未覆盖帧处理 | 所有窗口跑完后，统一向前后搜索填充 | 早期帧(0-11)`_frame_probs` 为空 → fallback 到后面帧的概率 |
| EMA | 在完整序列上逐类 EMA | 增量 EMA，初始值不同 |
| 结果 | 连续稳定 1→2→3→4→5 | 边界处偶尔跳变 |

**具体出错机制**（以 ok_8 为例，FSM路径 `[1,7,3,4,5,6]`）：

```
时间线（15fps，每帧 ≈ 0.067s）：

帧 0-11:  buffer 增长中，_frame_probs 全为空
         get_frame_prediction(10) → _frame_probs 中找不到 → fallback 向前搜索
         → 找到帧24的概率（S3:0.6, S2:0.3）→ 返回 S3
         → 但实际上帧10应该是 S1→S2 过渡期！

帧 12-23: 第1个窗口完成 (0-47)，_frame_probs[12-35] 被赋值
          EMA 开始，但起点已经被 fallback "污染"了

帧 24-35: 模型正确预测 S2
          但 FSM 已经因为之前报的 S3 进入了 ERROR 状态

结果: FSM 看到 S1→ERROR→S3（漏S2）→ 路径 [1,7,3,4,5,6]
```

**离线的 `sliding_window_inference()` 不会有这个问题**，因为：
1. 所有窗口一次性跑完
2. `_frame_probs` 全部赋值完后再统一处理未覆盖帧
3. EMA 在整个序列上一次性计算

### 3.5 影响评估

这个问题**不是模型的问题**，模型权重是对的。是**推理调度**的问题。

- 离线 eval 已经证明：同样的特征 + 同样的模型 + 同样的 FSM = 98.9%
- 实时 replay 只有 75% 是因为：特征提取是逐帧的（正确），但推理调度不一样（问题所在）

## 四、解决方案分析

### 方案A（推荐，最小风险）：回放用批量推理

**改造思路**：

修改 `replay_realtime_pipeline_v2.py` 的 `process_video()`：

```
第一遍遍历视频:
  - YOLO + MediaPipe（每帧）
  - 在 target_frame_indices 处提取90维特征
  - 存入 features 数组
  → 得到和离线 .npz 一样的特征矩阵 [N, 90]

第二遍（纯CPU，秒级）:
  - 调用 sliding_window_inference(features, model) → preds, confs
  - 逐帧把 preds 喂给 FSM
  → 得到 FSM 路径
```

**为什么这是正确的做法**：
- 回放测试的目的是验证"特征提取 + 模型 + FSM"这条链路是否在原始视频上也能工作
- 从原始视频提取的特征如果能和离线 .npz 一致，那么批量推理的结果必然一致
- **回放不需要真正"实时"，需要的是"正确"**

**改动范围**：
- 只改 `replay_realtime_pipeline_v2.py` 一个文件
- 不改 `TemporalPredictorV2`（回放不需要它）
- 不改 FSM
- 不改特征提取器

**预期结果**：
- 20 OK: 预期 19-20 通过（接近 98.9%）
- 20 WR: 预期 20 检出（100%）
- 如果达不到，说明特征提取环节有问题（而非模型或FSM）

### 方案B（后续，真正实时）：修复增量预测器

只有在方案A验证通过后才需要做。

**需要修复的点**：
1. 未覆盖帧不应 fallback，应返回 None → 上层 FSM 保持当前状态
2. 增量 EMA 需与批量 EMA 数学等价（可以缓存批量 EMA 值作为"正确答案"对照）
3. 第一个窗口完成前（前 T-1 帧），使用低置信度标记，FSM 不敏感

### 方案C（最后手段，不应轻易启动）：v3 特征重训

仅当方案A+B都验证通过但离线指标本身不满足需求时才考虑。
