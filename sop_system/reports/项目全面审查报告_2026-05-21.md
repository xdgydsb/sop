## 二、架构：Client-Server WebSocket 管线

### 2.1 实际运行架构

```
Windows 桌面                        Linux 4090 服务器
┌─────────────────┐                ┌──────────────────────────┐
│ hik_stream_client│  WS binary    │ inference_server.py      │
│ (Python 3.10)   │──────────────→│ (three_env conda)        │
│                 │ 12-byte头+JPEG│                          │
│ MVS SDK → 相机  │←──────────────│ Stage 1/2/3              │
│ BayerRG8→BGR    │  JSON结果     │ 端口8765                 │
│ 1280×720@15fps  │               │                          │
└─────────────────┘                └──────────────────────────┘
```

### 2.2 三个 Stage

| Stage | 功能 | 模块 | 用途 |
|-------|------|------|------|
| 1 | Echo | 仅解码+返回基本信息 | 验证网络链路 |
| 2 | YOLO检测 | YOLO + 粉红标记 + 回退检测 | 验证目标检测 + 调试充电器 |
| 3 | 全管线 | YOLO + MediaPipe + TCN+BiGRU + FSM | 完整SOP动作检测 |

### 2.3 二进制协议

```
帧格式: [frame_id:uint32 LE][timestamp:float64 LE][JPEG bytes]
响应:   JSON
```

**关键实现细节**:
- 使用 `websockets` 库 (asyncio server + sync client)
- 客户端用 MVS SDK Ctypes 绑定直接操作海康相机
- 服务器有帧跳过机制（`_processing` 锁），丢弃处理中到达的帧
- 支持 `--save-traces` 保存完整推理 trace（帧级 + 时序 + FSM）

## 五、推理管线详细分析

### 5.1 Stage 2（YOLO检测）— 当前主要使用

**流程**:
```
JPEG解码 → YOLO检测 (conf=0.20, imgsz=512)
  → ByteTrack追踪 (EMA bbox平滑 α=0.35)
  → 充电器假阳性过滤 (与box IoU>0.15 → 剔除)
  → 粉红标记辅助检测 (仅在YOLO未找到充电器时)
      ├── HSV粉色检测 → 裁剪ROI → YOLO重检测(conf=0.10)
      │   └── 成功 → pink_guided (conf×1.3)
      └── 失败 → expand_bbox → pink_marker (conf=0.75)
  → 颜色验证 (green_bag绿色验证, charger白色验证)
  → 每类置信度阈值过滤 + 迟滞(hysteresis 15帧)
  → FallbackDetector (边缘检测充电器 + HSV绿袋检测)
  → JSON输出
```

**迟滞机制**（防闪烁）:
- 正常阈值: charger=0.18, green_bag=0.18, earphone=0.25, box=0.25
- 低阈值（类刚被看到时）: charger=0.10, green_bag=0.10, earphone=0.15, box=0.12
- 15帧内该类被看到→使用低阈值；15帧未看到→恢复正常阈值

### 5.2 Stage 3（全管线）— 当前较少使用

**流程**:
```
同Stage 2的YOLO+粉红标记
  + MediaPipe 手部检测 (21点关键点×双手)
  + FeatureExtractorV2Standalone (15fps采样, 90维特征)
  + TemporalPredictorV2 (T=48, stride=4, 中心帧输出)
      ├── 累积buffer → 窗口推理 (TCN+BiGRU)
      ├── 多窗口概率平均 (_frame_probs)
      ├── 因果EMA平滑 (α=0.35)
      └── get_latest_mature_prediction() → {step, confidence, top3}
  + FSM验证 (合法转移: IDLE→S1→S2→S3→S4→S5→COMPLETE)
      ├── 低置信度(<0.4) → 保持当前状态
      ├── 合法转移 → 步骤推进
      ├── 乱序 → WRONG_ORDER 错误
      ├── 漏步 → MISSING_STEP 错误
      └── 超时(90s) → TIMEOUT 错误
  + Post-S5 自动完成 (1.5s后 → COMPLETE)
```

**已知架构问题**:

1. **物理状态模块未接入**: `physical_state.py` 已实现（盒子状态EMA、物体放置多阶段检测），但在 `SOPServer.process()` 中未真正使用，`physical_state_ok=True` 写死
2. **FSM 的 auto-recovery 是临时方案**: 当 FSM 进入 ERROR 状态且置信度 >0.8 时自动恢复到前一步。这可能在真实场景中掩盖真实的顺序错误
3. **idle-skip hack**: Stage 3 中当模型输出 S0 且当前在 S5 时直接跳到 COMPLETE。这在 `camera_smoke_test_v2.py` 中存在，但在 `inference_server.py` 的 SOPServer 中用了更优雅的 post-S5 delay 方案

### 5.3 粉红标记检测器 (PinkMarkerDetector)

**背景**: YOLO 无法可靠检测白色充电器（conf=0.0），作为辅助方案在充电器上贴粉红纸条。

**实现**:
- HSV 双范围: [0,19,32]~[30,255,255] + [150,19,32]~[180,255,255]
- 形态学: open(kernel=3) + close(kernel=9)，用于桥接手写文字造成的间隙
- 最小面积: 30px
- 2帧稳定性要求
- bbox 扩展比率: expand_w=1.8, expand_h=3.1（充电器比粉红纸条大）
- 支持从 `reports/pink_hsv_config.json` 加载标定配置

**三种充电器来源**:
| method | 含义 | 颜色标识 (客户端) |
|--------|------|------------------|
| `yolo` | YOLO直接检测到 | 黄色 |
| `pink_guided` | 粉红引导裁剪后YOLO重检测到 | 浅紫色 |
| `pink_marker` | 纯粉红标记回退(conf=0.75) | 品红 |

**真实现状**: 在实际测试中，`pink_guided` 从未成功过 —— YOLO 在粉红引导的裁剪区域内仍然无法识别充电器。`pink_marker` 回退方案是当前唯一可靠的充电器检测方式。

### 5.4 FallbackDetector

**白插头检测** (边缘+Canny):
- CLAHE 对比度增强 → Canny 边缘检测 → 轮廓查找
- 筛选条件: 面积>500px, 长宽比1.2-3.5, 平均亮度>130
- 置信度上限 0.55

**绿袋检测** (HSV):
- HSV 范围: [35,25,25]~[90,255,255]
- 形态学去噪
- 排除盒子内部区域
- 置信度上限 0.60

**问题**: 这两个回退方案在实际测试中也被证明效果有限。在困难样本目录中会保存漏检/低置信度帧供重训练。

## 七、代码质量评估

### 7.1 优点

- **模块化设计**: YOLO、MediaPipe、时序模型、FSM 各模块独立，接口清晰
- **良好的调试支持**: Stage 2/3 都有诊断打印（每30帧）、debug 图像保存、trace 日志
- **标定工具链完整**: 交互式 HSV 标定工具 (`calibrate_pink_interactive.py`) 可重复使用
- **防闪烁机制**: 迟滞 (hysteresis) 在两个 Stage 中一致实现
- **安全意识**: 配置中的敏感值从环境变量读取

### 7.2 需要改进

- **代码重复**: Stage 2 和 Stage 3 有大量重复代码（粉红标记检测、迟滞过滤、FallbackDetector 调用等），约 200 行重复逻辑
- **废弃代码未清理**: `fusion.py`, `temporal_lstm.py`, `hmm_filter.py` 标记为废弃但仍保留；三个已废弃模型仍在 models/ 目录
- **Import 位置不规范**: `from engine.yolo_detector import Detection` 在函数体内（为规避循环导入或作用域问题），容易导致后续 NameError
- **配置文件管理**: `config.py` 中的阈值与 `inference_server.py` 中的硬编码阈值存在重复定义
- **错误处理**: 多处 `except Exception: pass`，生产环境需要更严格的错误处理

## 十一、总结

### 什么在正常工作
- YOLO 对 box_open/box_closed, earphone 的检测可靠
- ByteTrack 追踪 + EMA bbox 平滑稳定
- 粉红标记 HSV 检测 + bbox 扩展回退方案已验证可用
- 迟滞机制有效防止 box_closed 闪烁
- WebSocket 通信链路、客户端 MVS SDK 集成均已验证

### 什么不工作
- YOLO 完全无法检测白色充电器（根本问题）
- pink_guided 重检测从未成功（裁剪策略可能需要调整）
- green_bag 检测存在可疑的 conf=1.0 假阳性
- Stage 3 全管线未经过真机测试
- FallbackDetector 的边缘检测方案在白背景下效果差

### 核心风险
**训练数据全部来自手机拍摄，当前模型在海康相机画面上的表现完全未知。** 这是整个项目最大的不确定性来源。

### 建议的下一步
1. 立即连接海康相机跑 Stage 2，收集 5-10 分钟真实画面
2. 分析 YOLO 在真实画面上的检测率（尤其是 earphone 和 green_bag）
3. 基于真实画面决定：是收集更多数据重训，还是现有模型已足够
4. 在 Stage 2 验证通过前，不要投入 Stage 3
