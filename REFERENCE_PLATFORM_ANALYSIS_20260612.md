# 参考视频平台化分析与实施蓝图

日期：2026-06-12  
参考文件：`参考.mp4`（约 43 秒，720x1280，30 FPS）

## 0. 产品原则（最高优先级）

本项目的目标是交付一个与参考视频同类、能够真实使用和继续扩展的工业 AI SOP 产品，不是给旧项目换皮，也不是必须在旧架构上修修补补。

现有程序只作为候选资产处理：

- 有价值且经过验证的能力可以复用，但必须通过清晰接口接入新产品。
- 功能正确但耦合严重的模块，先封装隔离，再逐步替换。
- 效果差、性能差、逻辑混乱、无法测试或阻碍产品目标的代码，直接废弃。
- 不因为旧数据库、旧页面、旧接口或旧技术选型而降低新产品需求。
- 不以“复用了多少旧代码”衡量进展，只以用户流程、检测效果、稳定性和可维护性验收。

每个旧模块复用前必须回答五个问题：

1. 现场效果是否经过真实相机验证？
2. 性能和延迟是否满足目标？
3. 输入、输出和失败状态是否清晰？
4. 是否能被自动测试和独立替换？
5. 复用成本是否低于重新实现，并且不会形成长期负担？

任意关键项不满足，就不能直接进入新产品主链路。

## 1. 结论

参考视频展示的不是单个固定 SOP 检测页面，而是一套“数据采集、模型训练、流程编排、设备下发、实时监测、异常追溯”闭环平台。

本项目可以向这个方向建设。以下已有能力只有通过上述复用评估后，才作为第一套算法或流程模板接入平台：

- 海康 MV-CS050-10UC 实时相机接入。
- 本地采集、服务器 GPU 推理的运行拓扑。
- stage3 五步包装 SOP 检测与时序判定。
- 开始、停止、重置、步骤状态、事件、告警和动作视频段。
- 若依前后端已有产品、流程、步骤、任务、监控、事件和告警基础页面。

当前缺少的是完整的产品架构和平台化配置能力。实施时先保护已经验证可用的检测基线，但不把 stage3 或现有若依实现视为不可替换的前提；它们可以作为首个适配器加速验证，最终是否保留由效果和工程质量决定。

## 2. 视频逐段理解

| 时间 | 视频内容 | 对应产品能力 |
|---|---|---|
| 0-8 秒 | 宣传定位、三步建立工业 SOP 模型 | 平台首页、产品价值说明 |
| 8-15 秒 | 上传视频/图片、样本分类、AI 自动标注、标签管理 | 数据集、样本、标注、训练 |
| 16-20 秒 | 拖拽流程节点，给节点绑定模型、标签和置信度 | SOP 可视化编排、算法绑定 |
| 20-25 秒 | 算力盒子/GPU 服务器、相机和任务状态 | 设备管理、模型下发、健康监控 |
| 26-31 秒 | 配置摄像头、视频流地址、预览并开始监测 | 相机管理、任务部署 |
| 32-43 秒 | 实时目标/手部识别、步骤推进、漏装错序报警 | 在线检测、流程判定、告警与统计 |

视频运行页不是只依赖 YOLO 单帧结果。它至少组合了：

- 目标检测：纸盒、保护件、设备、说明书、螺丝等。
- 手部或人体关键点：用于确认“由人执行了动作”，降低静态摆放误判。
- 空间关系：物体是否由盒外进入盒内。
- 时间关系：动作是否真实发生并保持稳定，而不是瞬时遮挡。
- 流程状态机：只接受当前步骤允许的事件，识别漏装、错装和顺序颠倒。

这与当前项目“YOLO + stage3 时序事件 + SOP 状态机”的正确方向一致。

## 3. 视频宣传与真实工程的差异

“无需算法工程师”和“AI 自动完成标注”是营销表达，不能直接作为交付承诺。真正可落地的定义应当是：

- AI 辅助标注，人员抽检和修正，而不是完全无人审核。
- 用配置降低重复开发，不代表新物料、新视角、新动作无需采样和验证。
- 每个模型必须有版本、验证集、准确率、召回率和现场验收记录。
- 流程节点必须支持连续帧、保持时间、进入/离开区域、动作前后状态等时序规则，不能只配置一个标签和置信度。
- 模型下发后必须有版本校验、回滚、设备心跳和运行日志。

## 4. 目标产品结构

产品的核心业务应定义为“工业工位作业合规与防错”：针对工单和产品型号，实时判断工人在指定工位执行的动作工序是否正确，并留下可以解释和追溯的证据。目标检测、手部关键点和时序模型是判定手段，不是最终产品结果。

完整领域定义见 `platform/docs/DOMAIN_MODEL.md`。

### 4.1 数据与模型中心

- 数据集：视频/图片导入、从视频抽帧、样本分类和去重。
- 标注任务：标签定义、人工标注、现有模型预标注、审核。
- 训练任务：训练参数、进度、日志、指标和产物。
- 模型仓库：模型版本、标签集、输入尺寸、运行环境、发布和回滚。
- 算法模板：目标检测、手部关键点、OCR、区域关系、计数、方向和时序事件。

### 4.2 SOP 设计中心

- 可视化流程画布：开始、步骤、判断、异常、结束节点。
- 步骤绑定：算法模型、目标标签、ROI、置信度、连续帧和保持时间。
- 时序规则：由外到内、由关到开、由开到关、目标留存、超时和错序。
- 流程版本：草稿、发布、停用、复制、版本对比和回滚。
- 仿真检查：发布前检查缺失模型、无出口节点、冲突规则和设备不兼容。

### 4.3 设备与部署中心

- 摄像头：海康 USB、RTSP/WebRTC 等来源、分辨率、帧率、预览和在线状态。
- 算力节点：本地采集端、边缘盒子、GPU 服务器、CPU/GPU/内存和心跳。
- 部署任务：将 SOP 版本、模型版本和相机绑定到工位。
- 运行控制：启动、停止、重置、重连和故障恢复。
- 版本一致性：前端显示的流程版本必须与推理服务实际运行版本一致。

### 4.4 运行与追溯中心

- 实时画面：低延迟视频、检测框、FPS、网络延迟、模型版本。
- 步骤面板：未开始、执行中、通过、失败、跳过和超时。
- 秒级告警：漏装、错装、错序、提前关盒、动作超时。
- 证据链：每一步对应截图、动作前后视频段、判定依据和置信度。
- 统计看板：完成数、通过率、告警数、平均节拍和工位健康度。

## 6. 推荐技术架构

保持现有前后端和检测链路，新增平台控制面：

```text
若依 Web 控制台
  ├─ 数据/模型中心
  ├─ SOP 设计中心
  ├─ 设备/部署中心
  └─ 实时监控与追溯
          │ REST / WebSocket
若依业务后端（配置、版本、任务、记录）
          │
运行调度层（发布配置、心跳、状态同步）
          │
采集 Worker ── 海康 USB / RTSP 相机
          │ 实时帧
推理 Worker ── stage3 / 后续模型适配器
          │ 结构化事件
通用 SOP Runtime ── 时序规则、状态机、证据视频段
```

关键边界：

- `stage3` 只负责它已经擅长的检测和事件输出。
- 平台通过统一 `AlgorithmAdapter` 调用 stage3，不直接侵入其内部实现。
- 通用 Runtime 消费结构化事件，不从页面状态反推检测结果。
- 页面只是展示和发控制命令，不能写死“某一步直接通过”。
- 配置发布后生成不可变版本，运行任务绑定明确版本，避免页面配置与检测服务脱节。

## 7. 分阶段实施

### 阶段 A：产品化演示版

目标：先达到老板视频中“看起来和用起来像平台”，同时保持当前检测效果。

- 冻结稳定检测版本并建立自动回归清单。
- 新增算法模型页，把当前 stage3 注册为第一个算法版本。
- 新增相机和算力节点页，展示本地采集端与远程 GPU 的在线状态。
- 新增 SOP 设计页，用可视化节点配置当前五步流程。
- 发布时将画布转换成明确的流程 JSON，并绑定模型版本。
- 重构监控页信息布局，不重构检测逻辑；补充设备、模型、延迟和版本信息。
- 将步骤截图、视频段、判定信息、告警统一为任务证据链。

验收标准：

- 现有正确/错误动作测试结果不得低于稳定基线。
- 页面可以完成“选模型、编流程、绑相机、发布、运行、追溯”的完整演示。
- 相机断开、推理服务断开必须明确报错并可恢复，不能黑屏无提示。
- 每个通过步骤都有对应动作视频段，不能用同一帧伪造多个步骤。

### 阶段 B：数据与模型闭环

- 视频导入和可配置抽帧。
- 标签集与标注任务。
- stage3/YOLO 预标注与人工审核。
- 训练任务、指标、模型版本和一键发布。
- 新模型必须先通过离线验证和现场小流量验证，再替换生产版本。

### 阶段 C：通用工业部署

- 多工位、多相机、多 SOP。
- 边缘 Worker 自动注册、模型下发、断点续传和回滚。
- 权限、审计、租户/工厂/产线层级。
- 生产统计、质量报表、节拍分析和告警闭环。

## 8. 第一轮具体开发顺序

1. 不改检测核心，建立稳定基线的启动、接口、实时帧和状态回归脚本。
2. 定义 `算法版本、流程版本、算力节点、摄像头、部署任务` 五类核心数据结构。
3. 先做只支持当前五步模板的 SOP 画布，验证配置到运行的闭环。
4. 将当前 bridge 和 stage3 包装为 Worker/Adapter，补心跳与版本上报。
5. 改造监控页和证据链，让演示效果达到参考视频的完整度。
6. 基线测试通过后，再进入数据集、标注和训练工作台。

## 9. 明确禁止的做法

- 不在新页面开发时顺手修改 stage3 检测规则。
- 不用离线视频替代真实相机验证。
- 不把单帧出现某标签等同于动作完成。
- 不用前端死代码直接将步骤置为通过。
- 不先做漂亮空壳，再补检测和视频证据。
- 不在同一工作目录里复制多套互相抢端口的启动程序。
- 不发布无法回滚、无法识别模型版本的更新。

## 11. 新项目代码组织

- 新产品代码统一放入 `platform/`，旧目录不得继续承载新功能。
- 采用“Web、业务 API、SOP Runtime、采集/推理 Worker、共享契约、基础设施”分层。
- 新代码禁止直接引用 `sop_system/` 或 `SOP前后端/`；复用能力必须经过 legacy adapter 转换成统一契约。
- 先按完整纵向能力开发，再进入下一个模块，禁止同时铺开大量静态页面和空接口。
- 当前架构规则见 `platform/docs/ARCHITECTURE.md`。
