## 已纠正的误导状态

- 平台配置写入不再显示为真实 `ACTIVE` 部署，而是 `STAGED`。
- 只有 Worker/Runtime 返回应用确认后，未来才能显示 `APPLIED` 或 `ACTIVE`。
- stage3 标记为经过真实相机验证的稳定基线，不虚构训练指标。
- 未实现的数据训练按钮不得作为可用功能宣传。

## 旧资源使用结论

- `sop_system/ruoyi_hik_ws_bridge.py`：`WRAP`。继续提供海康实时流、stage3
  推理、步骤截图和动作片段；通过 Runtime 接口接入新平台。
- 若依 Runtime 接口：`WRAP`。继续提供会话控制和当前步骤状态。
- 旧若依检测页面：`REPLACE`。保留作稳定回归参考，不作为新产品界面基础。
- stage3 检测核心：`REUSE`。保持稳定基线，除非真实相机回归证明存在具体缺陷。
- 旧 CRUD 页面：按领域数据逐项评估，不批量搬入新平台。

## 官方资源使用决策

- 数据标注优先评估 CVAT 集成。CVAT 官方支持图片、视频、跟踪以及自托管模型
  辅助标注，适合作为成熟标注能力而不是重新开发低质量标注器：
  <https://docs.cvat.ai/docs/annotation/auto-annotation/automatic-annotation/>
- 物体由盒外进入盒内需要稳定目标身份时，优先在独立视觉适配器中评估
  Ultralytics Track 模式的 ByteTrack/BoT-SORT，不侵入 SOP Runtime：
  <https://docs.ultralytics.com/modes/track>
- 多工位或多相机性能成为真实瓶颈后，再评估 NVIDIA DeepStream 的流式推理和
  跟踪组件；当前单工位不为技术炫耀提前迁移：
  <https://docs.nvidia.com/metropolis/deepstream/dev-guide/text/DS_Overview.html>

## 下一阶段严格优先级

1. 为部署增加 Worker 接收、版本校验、应用确认和回滚。
2. 在真实运行页显示采集序号、队列延迟、推理延迟和端到端延迟。
3. 完成任务历史、每步截图/视频段检索以及系统中断记录。
4. 接入数据导入与 CVAT 标注任务，而不是自研完整标注器。
5. 有真实快速动作漏检数据后，再评估跟踪器或动作模型。
