# 平台界面与交互要求

## 1. 界面是产品主体

本项目最终交付形态是浏览器中的工业 AI SOP 平台。OpenCV、YOLO、跟踪器、Runtime 和 Worker 都是后台能力，用户不需要直接操作脚本、终端或算法代码。

平台不得退化为：

- 只能通过命令行启动和操作的算法程序。
- 只有实时视频、没有配置和追溯能力的单页工具。
- 只有静态页面、按钮不连接真实业务的演示壳。
- 把旧若依 CRUD 页面简单换色后称为新产品。

## 2. 一级产品入口

### 工作台

- 今日工单、运行工位、产量、一次通过率和告警。
- 相机、Worker、GPU和模型服务健康状态。
- 当前异常和需要人工复核的任务。

### SOP设计

- 产品与SOP列表。
- 可视化工序编排。
- 每个工序的前置、动作、后置、错误和证据规则。
- 模型、标签、ROI、时间窗口和阈值绑定。
- 草稿、验证、发布、版本比较和回滚。

### 数据与模型

- 视频和图片数据集。
- 抽帧、标注、审核和数据版本。
- 训练任务、评估指标和模型版本。
- 模型发布状态及其适用工序。

### 工位与设备

- 产线、工位、相机和算力节点。
- 实时预览、ROI校准、帧率和延迟测试。
- SOP/模型部署和版本一致性。
- 在线、离线、异常和恢复状态。

### 实时检测

- 真实低延迟相机画面。
- 当前工单、产品、操作员、工位和SOP版本。
- 工序步骤、当前动作、判定进度和错误提示。
- 开始、停止、重置功能必须连接真实Runtime。
- FPS、队列、推理延迟和端到端延迟。
- 当前步骤的判断依据，不能只显示一个结果颜色。

### 任务与证据

- 每件产品的完整作业周期。
- 每一步的开始、结束、结论和判定依据。
- 对应截图和动作前后视频段。
- 漏做、错序、错件、超时、无法判断和系统中断。
- 人工复核、返工和处理记录。

## 3. 实时检测页硬性要求

- 页面打开后显示实时相机流，不使用离线视频或过时静态图片代替。
- 画面与判定状态使用同一采集序号或时间戳关联。
- 页面明确显示采集FPS、推理FPS、排队延迟和判定延迟。
- 当相机、Worker或推理服务断开时，显示具体故障和恢复状态。
- 系统故障不得显示为工人操作失败。
- 开始、停止、重置是幂等操作，一次点击必须得到明确结果。
- 重置后步骤、告警、动作候选和证据缓存必须一致清空。
- 页面刷新后从后端恢复真实会话，不能重新创造一套前端假状态。
- 每个通过或失败结果都能打开对应证据，证据必须匹配实际动作。

## 4. 界面质量

- 面向工厂大屏和普通电脑，优先支持1920x1080，同时适配较小屏幕。
- 实时运行页采用高信息密度，但当前步骤和告警必须一眼可见。
- 配置页保持清晰分区，不在一个表单堆叠所有算法参数。
- 技术参数提供合理默认值和解释，普通业务用户无需理解代码。
- 危险操作、发布和回滚需要确认及审计记录。
- 全站统一使用中文业务术语，内部模型字段不得直接暴露给用户。

## 5. 前后端同步交付规则

从下一条产品能力开始，采用纵向切片：

```text
领域契约
  -> 后端行为
  -> Worker/Runtime接入
  -> 平台界面
  -> 浏览器端到端验收
```

不允许连续开发大量后台模块后才补界面，也不允许先铺大量页面后再补真实功能。

首个界面切片为“单工位实时检测”：

1. 选择已发布SOP、工位和相机。
2. 完成相机、Worker和推理服务自检。
3. 开始一个真实作业周期。
4. 显示实时画面、五步进度和判定依据。
5. 显示快速动作关键帧和端到端延迟指标。
6. 停止、重置并查看步骤证据。

## 6. 验收方式

- 每次显著界面变更必须在真实浏览器打开验证。
- 检查加载、空数据、故障、恢复和正常运行状态。
- 控制台不得有未处理错误。
- 按钮必须对应真实接口和真实状态变化。
- 用户流程由浏览器端到端测试覆盖。
- 后台单元测试通过不代表平台功能完成。
