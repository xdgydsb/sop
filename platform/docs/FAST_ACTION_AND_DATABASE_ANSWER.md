# 快速动作与动态 SOP 问题说明

日期：2026-06-13

## 一、为什么导师快速操作时系统没有反应

问题不是单纯“GPU不够快”，而是整条链路的采样和排队共同造成的。

旧链路实测/默认状态：

- 相机本地约 23 FPS。
- bridge 默认最多发送 20 FPS。
- stage3 页面曾显示服务器约 12 FPS。
- WebSocket 服务在同一个接收循环中同步执行 GPU 推理。
- 多个稳定器和事件规则仍使用连续帧数，而不是毫秒。

如果一个关键动作只持续 180 ms：

- 20 FPS 理论上只有约 3.6 个采集帧。
- 12 FPS 理论上只有约 2.16 个推理帧。
- 如果队列积压 6 帧，12 FPS 下最坏排队时间约 500 ms，已经远长于动作本身。

所以快速动作可能没有留下足够的“动作前、动作中、动作后”证据，或者证据到达时
已经严重滞后。简单降低置信度、把连续帧改成 1 帧，虽然可能增加召回，但会重新引入
遮挡误判、闭盒上放物误判和单帧抖动误判。

## 二、正确的提速顺序

1. 相机尽量以 30-60 FPS 采集，曝光时间要足够短，避免运动模糊。
2. 采集线程与推理解耦，使用容量 2-3 帧的有界队列，禁止无限积压。
3. 每个采集帧运行低成本 ROI 运动评分，动作关键帧优先进入 GPU。
4. GPU不必平均处理每帧，但必须保留动作前、交互、跨区和动作后关键帧。
5. 把“连续 N 帧”逐步改为真实毫秒窗口，避免 FPS 变化改变业务语义。
6. 用 ByteTrack/BoT-SORT 等跟踪器保持物体身份，允许手部造成的短时遮挡。
7. 使用 TensorRT FP16；在准确率验证通过后再评估 INT8。
8. 对慢、中、快三档动作分别统计每步召回率、误报率和 P95 端到端延迟。

当前新平台已经实现：

- ROI 运动关键帧评分。
- 关键帧优先的有界队列。
- 1200 ms 短时交互/状态证据锁存。
- 真实时间戳和流水线延迟统计。
- 快速动作性能预算计算和自动测试。

尚需现场完成：

- 把上述 Worker 正式接到海康桥接和服务器 stage3 前面。
- 将服务器 WebSocket 接收与 GPU 消费改为独立协程/线程。
- 采集 30/45/60 FPS 与推理 20/25/30 FPS 的真实对比数据。
- 在服务器导出 TensorRT 引擎并验证检测准确率没有下降。

## 三、边缘设备到底为什么可能更快

边缘设备不是“放一个盒子就自动变快”。它的价值是把推理放到相机和工位附近：

- 避免每帧 JPEG 编码、网络上传、服务器排队和结果回传。
- 网络中断时仍能检测。
- 多工位时每个工位有独立算力预算，避免共享服务器互相排队。
- 可直接使用相机原始帧，减少重复压缩和拷贝。

但本项目已有局域网 4090 服务器。单工位情况下，4090 的纯推理通常比中低端边缘盒子
更快；当前主要瓶颈更可能是串行传输、队列和规则的帧数阈值。因此正确路线是先测量并
修复软件流水线，再决定是否采购 Jetson Orin NX/AGX Orin 或工业 GPU 边缘盒子。

海康官方页面显示 MV-CS050-10UC 在 2448x2048 下最高可达 60 FPS，因此当前约
20 FPS 不是相机型号的理论上限，需要继续检查曝光、像素格式、USB3 带宽和 bridge
发送限制：
<https://www.hikrobotics.com/en/machinevision/visionproduct/?id=134&typeId=78>

NVIDIA 官方说明 TensorRT 的目标就是低延迟、高吞吐生产推理，并要求先基准测试、
再优化、再复测：
<https://docs.nvidia.com/deeplearning/tensorrt/latest/performance/best-practices.html>

Ultralytics 官方 Track 模式支持 ByteTrack 和 BoT-SORT，可为跨帧对象提供持续 ID：
<https://docs.ultralytics.com/modes/track>

## 四、为什么删除步骤后检测页仍显示五步

旧系统存在两个数据源：

1. MySQL `sop_step` 表保存页面编辑的步骤。
2. `ruoyi_hik_ws_bridge.py` 写死了五个事件、五个名称、`range(1, 6)` 和最大步骤 5。

若依后端创建任务时会从 `sop_step` 生成任务步骤快照，但 bridge 随后仍持续发送固定
五步运行数据。更重要的是 stage3 本身的 `EVENT_SEQUENCE` 和状态机也是固定五步。
因此删除数据库记录并不等于算法自动变成四步，旧页面看起来就像“假改动”。

## 五、新平台的数据库处理

新平台已经改为本地 SQLite，而不是 JSON：

- `model_registry`：模型登记和验证状态。
- `workstation`：工位、相机和推理节点。
- `sop_draft`：当前可编辑草稿。
- `sop_draft_step`：按顺序保存任意数量步骤。
- `sop_release`：不可变发布快照。
- `deployment`：发布版本到工位的部署记录。

数据库文件默认位于：

```text
runtime_logs/platform/platform.db
```

删除步骤并保存后，重新读取草稿会真实变更。发布时系统会编译兼容性：

- 事件序列仍是标准五步：`LEGACY_STAGE3`。
- 任意增删、改序或更换事件：`GENERIC_RUNTIME_REQUIRED`。

不兼容版本可以保存和发布，但部署状态为 `BLOCKED`，直到通用 Runtime 和对应视觉
事件适配器真正支持该流程。这比数据库改了、检测端仍偷偷执行固定五步更可靠。

## 六、导师面前可以如何回答

可以简洁说明：

> 当前快速动作漏检主要不是 YOLO 单点问题，而是 20 FPS 发送、约 12 FPS 推理、
> 串行网络处理、帧数型稳定规则共同导致的时域欠采样和排队。边缘设备能减少网络与
> 共享服务器排队，但不是唯一答案。我们先采用高帧率采集、关键帧优先有界队列、
> 跟踪与毫秒时序窗口，并用端到端 P95 延迟和快动作召回率验收；再根据测量结果选择
> 4090 服务器 TensorRT 或 Jetson 工业边缘盒子。流程配置方面改为草稿、不可变发布
> 和部署版本三层，数据库修改只有在 Runtime 确认兼容并应用后才进入生产。
